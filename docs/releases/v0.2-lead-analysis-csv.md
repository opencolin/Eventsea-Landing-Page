## Goal

Turn the operator-run event-leads skill into a self-serve SaaS surface:

1. **Sponsor uploads a registration CSV** (Luma export, Eventbrite, custom).
2. **Sponsor configures their ICP** (product + one-line description, multi-sponsor supported).
3. **Eventsea runs the full pipeline** — ingest → triage → two-tier verify → score per sponsor → influence rank → deliverables.
4. **Sponsor gets a private, password-gated dashboard** + downloadable spreadsheet + HubSpot/Salesforce CSV exports.

This is the *product* that the Sprint 1 pilots prove out via concierge fulfillment. Once 5+ pilots are running, v0.2 ships and the operator stops being the bottleneck.

Success metric: **the next 10 paid customers self-serve the pipeline end-to-end with zero operator touch on 90% of submissions.**

## Non-goals

- Not building Luma API ingestion yet. That's v1.0 (skill v1.1).
- Not building autonomous event discovery. That's v2.0 (skill v2).
- Not building the marketplace transactional layer. That's the original `release/v2.0-marketplace-tx`.

## Tech sketch

- **Frontend**: React (existing). New routes:
  - `/app/upload` — CSV upload + sponsor config
  - `/app/job/:id` — pipeline status, ETA, intermediate results
  - `/app/dashboard/:slug` — the rendered single-file dashboard (still server-side rendered for password gate)
  - `/app/exports/:id` — XLSX + CRM CSV downloads
- **Backend**: Express + Drizzle/Neon (per existing stack)
  - `POST /api/jobs` — accept CSV upload (multipart), persist, enqueue
  - `GET /api/jobs/:id/status` — polling endpoint
  - `GET /api/exports/:id?format=xlsx|hubspot|salesforce` — generate on demand from persisted JSON
- **Background pipeline**: Inngest workflows mirroring the skill's steps
  - Step 1 → ingest worker
  - Step 2 → fan-out screening (batch size ~12, parallel workers) + deep enrichment for top decile
  - Step 4 → social fetcher (rate-limited fxtwitter + gh api)
  - Step 5 → renderer (XLSX + HTML + CSV generation)
- **AI**: Claude API per `~/.claude/skills/claude-api` skill. Default to `claude-sonnet-4-6` for deep enrichment, `claude-haiku-4-5-20251001` for screening. Cache the system prompts.
- **Verification source**: Tavily for web search (matches the skill's named sponsor + the skill's "live web sources" approach).
- **Auth + multitenancy**: needed eventually but **OUT OF SCOPE for v0.2**. Initial release is per-org password-gated, owner-only access — same model the skill's "PII rule" describes.

## Hard rules from the skill (preserve in code)

- **LinkedIn URL/slug is the identity anchor.** Email domain + verbatim bio are self-verifying; web claims are not.
- **Name collisions = #1 hallucination source.** Reject when there's no anchor. Better "unscreenable" than "wrong company."
- **No invented funding figures.** Sources or "not found."
- **VERDICT cap discipline**: light scores cap at 8; U caps at 3; deep scores can hit 9.
- **Sponsor-employee emails**: scores zeroed, do-not-pitch flag.
- **Competitor conflict override**: employee/fund of a sponsor's competitor → that sponsor's score floored to 1 with explicit flag.
- **Investors score equally across sponsors** (partnership leverage, not consumption).
- **Builders and judges must be mutually exclusive** — assert in code.

## Acceptance test

A new sponsor signs up, uploads a Luma CSV of 250 attendees + their 3-product ICP. Within 30 minutes, they get an email with:
- A password-gated dashboard URL displaying ranked leads with per-sponsor score pills
- A downloadable XLSX with color-coded rows
- HubSpot + Salesforce CSV exports of the current view
- All without an operator touching the job
