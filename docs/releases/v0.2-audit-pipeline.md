# v0.2 — Make the wedge real: working calendar audit pipeline

> Status: **planned**. Spec is ready for an implementation agent to pick up.
> Branch: `release/v0.2-audit-pipeline`
> Estimated effort: 8–12 eng-days for one engineer
> Council reference: see `../council/product-pm-proposal.md` and `../council/decision.md`

## Goal

Today the landing page says "Paste your Luma calendar → AI audit in 3 minutes" and the form just stores the URL in memory. v0.2 makes that promise real: a submitted URL produces a real audit, emailed to the user within minutes.

Success metric: **the next 10 audit submissions all produce a real audit report delivered by email within 24 hours.**

## Non-goals (explicit)

- Not building a logged-in product UI yet. That waits for v1.0.
- Not building multi-tenant orgs/users yet. That waits for v1.0.
- Not building the team digest yet. That waits for v1.5.
- Not building sponsor matchmaking transactions. That waits for v2.0.

## In-scope features

### 1. Luma calendar scraper (server-side)
- Input: a public Luma calendar URL (`lu.ma/<slug>`)
- Output: a structured list of events with title, date, location, description, host, attendee count, sponsor list, speaker list
- Implementation: `server/scrapers/luma.ts`. Use `fetch` + HTML parsing (cheerio) or Luma's public API if available.
- Cache scraped results in the DB (next bullet) so re-audits are free.

### 2. Real database
- Stop using `MemStorage`. Wire `@neondatabase/serverless` + Drizzle to a real Neon instance.
- New tables on top of the existing ones (`shared/schema.ts`):
  - `scraped_calendars` (url, fetched_at, raw_json)
  - `scraped_events` (calendar_id, title, date, location, description, host, raw_json)
  - `audit_reports` (calendar_audit_id, generated_at, json_report, markdown_report, email_status)
- Drizzle migration with `drizzle-kit push`. Document the `DATABASE_URL` env var in `.env.example`.

### 3. AI audit generator
- Input: one `scraped_calendars` row + its `scraped_events`.
- Output: a structured `audit_reports.json_report` matching the deliverables shown on the landing page:
  - Activity score (0–100)
  - Sub-scores: Cadence, Audience Quality, Description Craft, Co-Sponsor Network
  - Per-event scores
  - Copy critique per event ("rename X to Y for +N% RSVPs")
  - Untapped co-sponsors (3–5 named companies with rationale)
  - Attendance breakdown (roles, repeat attendance)
  - Events to sponsor next (3–5 upcoming events with match scores)
- Implementation: `server/audit/generate.ts`. Use the Claude API skill (see CLAUDE.md / skill). Default to `claude-sonnet-4-6` for cost, escalate to `claude-opus-4-7` for the critique. Use prompt caching for the system prompt.
- Save BOTH JSON (for future product UI) AND Markdown (for email).

### 4. Background job pipeline
- When `POST /api/calendar-audit` lands:
  1. Insert calendar_audit row (already exists).
  2. Enqueue a job: `scrape → audit → email`.
  3. Return 202 with a polling URL.
- Implementation: BullMQ + Redis OR Inngest. Pick the simpler one for solo founder ops.
- Failure handling: retries with exponential backoff, dead-letter to admin.

### 5. Audit email
- Resend integration (already advertised in the landing page integrations grid).
- Template: hero with activity score, sub-scores grid, top 3 insights, "view full report" CTA pointing to a hosted public report page.
- `client/src/pages/report.tsx` — public report page at `/r/:reportId` that renders the markdown report. No login required.

### 6. Audit form UX upgrades
- `client/src/components/calendar-audit-form.tsx`: after submit, show "scraping your calendar..." → "scoring events..." → "writing your audit..." progress states by polling the audit status endpoint every 3s.
- If the user provides email upfront, skip the second-step modal.

## Risks + mitigations

- **Luma rate limits / scraping fragility** — mitigate by caching scrapes + degrading gracefully when a calendar can't be fetched ("we'll retry in 1 hour")
- **AI cost** — cap audits at 50 events per calendar; use Sonnet not Opus by default; enable prompt caching on the system prompt
- **Email deliverability** — set up SPF/DKIM on `eventsea.ai` before sending

## Acceptance test

A non-technical user pastes `lu.ma/frontier-tower-sf` into the form. Within 5 minutes (warm path) or 30 minutes (cold path), they get an email with:
- Activity score
- 6 deliverables (per the landing page promise)
- A link to a public report page that renders the same content
